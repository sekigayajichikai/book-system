# 自治会館予約管理 — UI方式の比較整理

## 現在ある5つの方式

| # | 方式 | 形態 | 状態 |
|---|------|------|------|
| 1 | HTML日程表（横型グリッド） | Web公開ページ（GitHub Pages） | 運用中 |
| 2 | HTMLプロトタイプ（入力モード付き） | ローカルHTML | プロトタイプ |
| 3 | スプレッドシート「確定」シート | Google Sheets（行形式） | 運用中 |
| 4 | スプレッドシート「グリッドシート」（縦型） | Google Sheets（日付=行、部屋=列） | 今回作成 |
| 5 | 予約アプリ（book-system） | React SPA（Vercel） | Phase1完成・未デプロイ |

---

## 方式別メリット・デメリット

### 1. HTML日程表（横型・Web公開）
**誰向け:** 住民（閲覧）、事務局（印刷・掲示）

| メリット | デメリット |
|----------|-----------|
| 住民がスマホ・PCで空き確認できる | 閲覧専用（入力不可） |
| 印刷して会館に掲示できる | 更新にはGAS実行が必要 |
| 従来の紙カレンダーと同じ見た目 | スマホでは横スクロールが必要（PC向き） |
| スマホ用カード表示もあり | |

### 2. HTMLプロトタイプ（入力モード付き横型グリッド）
**誰向け:** 事務局（入力・管理）

| メリット | デメリット |
|----------|-----------|
| 従来の紙カレンダーと同じ操作感 | ローカルのみ（Web未公開） |
| カテゴリ・団体のマスタ選択UI | データ保存先が未接続（GAS APIは用意済み） |
| キャンセル・変更の履歴管理 | 操作にはPC必要（スマホ非対応） |
| 料金自動計算・用具管理 | |

### 3. スプレッドシート「確定」シート（行形式）
**誰向け:** 事務局（データ管理の本体）

| メリット | デメリット |
|----------|-----------|
| データの正本（他の方式はここから生成） | 一覧性が低い（行が並ぶだけ） |
| フィルタ・検索・ソートが自由 | 「何日の何が空いているか」が見にくい |
| GASで自動生成・公開と連携済み | 手入力のミス（日付・部屋名の打ち間違い）リスク |
| 複数人で同時編集可能 | |

### 4. スプレッドシート「グリッドシート」（縦型）
**誰向け:** 事務局（一覧確認・簡易入力）

| メリット | デメリット |
|----------|-----------|
| スプレッドシート上で空き状況が一目瞭然 | 「確定」シートとの二重管理になる |
| セル直接編集できる（Sheets慣れた人向け） | 編集しても「確定」シートに反映されない |
| 部屋ごとの色分けで見やすい | GAS実行で毎回再生成（編集内容が消える） |
| 追加ツール不要（Sheetsだけで完結） | 印刷レイアウトは横型より劣る |

### 5. 予約アプリ（book-system）
**誰向け:** 住民（予約申込）、事務局（受付管理）

| メリット | デメリット |
|----------|-----------|
| 住民がスマホから直接予約できる | 未デプロイ（まだ使えない） |
| Googleカレンダー連携で空き状況リアルタイム | 事務局の承認フロー未実装 |
| LINE通知で事務局に即時連絡 | 既存のSheets運用との二重管理リスク |
| 24時間受付可能（窓口時間外でもOK） | 高齢者にはハードルがある可能性 |
| 将来LIFF対応でLINEから直接アクセス可能 | 保守・運用コストが追加で発生 |

## どう組み合わせるか（提案の軸）

### 最小構成（今すぐ運用できる）
```
事務局の入力: スプレッドシート「確定」シート（行形式）
  ↓ GAS自動生成
住民の閲覧:  HTML日程表（Web公開）+ 印刷掲示
```

### 事務局の入力改善（次のステップ）
```
事務局の入力: HTMLプロトタイプ → GAS API → 「確定」シート
  ↓ GAS自動生成
住民の閲覧:  HTML日程表（Web公開）+ 印刷掲示
```
※ グリッドシートはプロトタイプの代替案。どちらか一方でよい。

### フル構成（将来像）
```
住民の予約:   予約アプリ（book-system）→ Googleカレンダー
  ↓ 同期
事務局の管理: スプレッドシート or HTMLプロトタイプ
  ↓ GAS自動生成
住民の閲覧:  HTML日程表（Web公開）
```

---

## 判断ポイント

1. **グリッドシート（#4）は必要か？**
   → HTMLプロトタイプ（#2）がGAS APIに繋がれば不要になる可能性が高い

2. **予約アプリ（#5）はいつデプロイするか？**
   → 事務局の承認フローと既存Sheets運用との統合設計が先

3. **事務局の入力方法を統一するか？**
   → 現状「確定シート直接編集」。改善するならHTMLプロトタイプかグリッドシートのどちらかに絞る
