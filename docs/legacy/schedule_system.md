# 自治会 予定管理・Web公開システム — 実装指示書

宛先: Claude Code
依頼者: はな(自治会DX委員)
最終更新: 2026-04-11
バージョン: 2.0(A案・予定マスタ正規化モデル)

> **本指示書は前バージョン(自治会館空室マトリクスのWeb化)を全面改訂したものです。**
> 旧版の「マトリクスを直接編集する」前提から「予定マスタ(1イベント1行)を編集し、そこから複数のビューを生成する」設計に転換しています。

---

## 1. 目的

自治会の全予定(自治会館の予約、コミュニティハウスでの会議、屋外活動など)を**1つのスプレッドシート(予定マスタ)に集約**し、そこから:

- 自治会館の使用状況マトリクス表(従来Excelの見た目を踏襲)をHTML自動公開
- 主催団体別のGoogleカレンダー5個に自動同期
- 会場別(自治会館)のGoogleカレンダー1個に自動同期

を生成する。役員は予定マスタだけを編集すればよい。

---

## 2. 全体構成

```
[Googleスプレッドシート: 予定マスタ]   ← 役員が編集(1イベント=1行)
        ↓ (編集トリガー + 日次保険トリガー)
[Google Apps Script]
   ├→ HTML生成 → GitHub Pages公開(自治会館マトリクス表)
   ├→ Googleカレンダー同期(主催別5個)
   └→ Googleカレンダー同期(会場別1個)
        ↓
[住民・役員のスマホ/PC]                ← LINEリッチメニューから誘導
```

---

## 3. 入力: 予定マスタの構造

### 3.1 シート構成

- ファイル名: 「自治会予定マスタ」
- メインシート名: `events`
- 補助シート: `master_orgs`(主催団体マスタ)、`master_venues`(会場マスタ)

### 3.2 events シートの列定義

| 列 | 列名 | 型 | 必須 | 説明 |
|---|---|---|---|---|
| A | id | 文字列 | 自動 | GASが採番(例: `evt_20260412_001`) |
| B | 日付 | 日付 | ◯ | YYYY-MM-DD |
| C | 終日 | チェック | - | チェック時はD/Eを無視 |
| D | 開始時刻 | 時刻 | △ | 終日でなければ必須 |
| E | 終了時刻 | 時刻 | △ | 終日でなければ必須 |
| F | 主催団体 | プルダウン | ◯ | master_orgsから選択 |
| G | 会場 | プルダウン | ◯ | master_venuesから選択 |
| H | 部屋 | プルダウン | △ | 会場=自治会館の場合のみ必須 |
| I | 区分 | プルダウン | △ | 会場=自治会館の場合のみ必須(午前/午後/夜間) |
| J | イベント名 | 文字列 | ◯ | カレンダー・HTMLに表示される |
| K | 備考 | 文字列 | - | 説明文。カレンダーのdescriptionに入る |
| L | 更新日時 | 日時 | 自動 | GASが書き込む |

**自治会館を選んだときだけH/Iを必須化**するバリデーションをGASで実装(onEdit時にチェック、不足ならセル背景を赤く)。

### 3.3 master_orgs(主催団体マスタ)

| 主催団体名 | カレンダーID(Script Properties側で管理) |
|---|---|
| 役員 | GCAL_ID_YAKUIN |
| 事務局 | GCAL_ID_JIMUKYOKU |
| 委員会 | GCAL_ID_IINKAI |
| 地区長・班長 | GCAL_ID_CHIKU_HAN |
| 自主活動部 | GCAL_ID_JISHU |

例: 「DX委員会」「建築協定委員会」は主催団体=「委員会」、イベント名に「DX委員会」「建築協定委員会」と入れる。

### 3.4 master_venues(会場マスタ)

| 会場名 | 部屋必須 | 備考 |
|---|---|---|
| 自治会館 | ◯ | 部屋・区分の入力が必須 |
| コミュニティハウス | - | 部屋区分なし |
| 屋外/その他 | - | イベント名に場所詳細を書く(例:「山歩き(高尾山)」) |

自治会館の部屋:
- 会議室
- 和室(畳側)
- 和室(椅子側)
- 図書室

### 3.5 時間区分(自治会館専用、参考)

| 区分 | 基本時間 | 延長時 |
|---|---|---|
| 午前 | 9:00-12:00 | 〜13:00 |
| 午後 | 13:00-16:00 | 〜17:00 |
| 夜間 | 17:00-20:00 | 〜21:00 |

自治会館の予定を入力するときは、開始/終了時刻を区分の基本時間で自動補完する(D/Eが空のまま区分が選ばれたら、GASがonEditで埋める)。役員が手動で上書きしてもよい。

## 4. 出力1: 自治会館マトリクスHTML

### 4.1 仕様

- 元のExcel様式を踏襲: 縦軸=部屋×時間帯、横軸=日付、セル=イベント名
- データソース: events シートから `会場 = 自治会館` のレコードを抽出
- グルーピング: `日付 × 部屋 × 区分` の3次元
- レンダリング先: GitHub Pagesの `kaikan/YYYY-MM.html`

### 4.2 デザイン要件(旧指示書から継承)

- PC: マトリクス一望
- スマホ: 横スクロール + 768px未満では「日付タップ→当日の予定一覧」のドリルダウン
- 高齢者配慮: 文字サイズ最低14px(スマホ16px以上)、今日の列ハイライト、最終更新日時を上部に大きく表示
- React等は使わない、素のHTML+CSS+最小限のvanilla JS、1ファイル完結
- 凡例に「午前=9:00-12:00 / 午後=13:00-16:00 / 夜間=17:00-20:00」を表記

### 4.3 ファイル構成

```
kaikan/
├── index.html       ← 当月へリダイレクト
├── 2026-04.html
├── 2026-05.html
└── (style.cssはインライン化推奨)
```

## 5. 出力2: Googleカレンダー(計6個)

### 5.1 カレンダー一覧

**主催別(5個)**: events シートを `主催団体` でフィルタ

| カレンダー名 | フィルタ条件 |
|---|---|
| 自治会 - 役員 | 主催団体 = 役員 |
| 自治会 - 事務局 | 主催団体 = 事務局 |
| 自治会 - 委員会 | 主催団体 = 委員会 |
| 自治会 - 地区長・班長 | 主催団体 = 地区長・班長 |
| 自治会 - 自主活動部 | 主催団体 = 自主活動部 |

**会場別(1個)**: events シートを `会場` でフィルタ

| カレンダー名 | フィルタ条件 |
|---|---|
| 自治会館 - 使用状況 | 会場 = 自治会館 |

### 5.2 イベントの作り方

| 項目 | 値 |
|---|---|
| タイトル | events.イベント名(自治会館使用状況カレンダーのみ `[部屋名]イベント名` のプレフィックス付き) |
| 開始/終了 | events.開始時刻/終了時刻 (終日フラグ on なら終日イベント) |
| 説明 | events.備考 + 改行 + `[auto-generated by jichikai-sync] id=evt_xxx` |
| 場所 | events.会場 (+ 部屋があれば連結) |

**重要**: 同じ events 1行が、主催別カレンダーと会場別カレンダーの両方に登録される場合がある(例: 自治会館での総会 → 「役員」と「自治会館使用状況」の両方に出る)。これは設計として正しい。

### 5.3 同期方式: 全削除→全再作成

旧指示書 Section 11.4 と同じ方針:

1. 対象月の全イベントをカレンダーから取得
2. **本GASが作成したイベントだけ**を削除(`description` の `[auto-generated by jichikai-sync]` で判定)
3. events シートを読み直して全イベントを再作成

冪等性を担保し、差分同期のバグを避ける。

### 5.4 役員向け運用ルール(README必須記載・強調)

> **Googleカレンダーは見るだけです。直接編集しないでください。**
> 編集してもスプレッドシートが次に更新されたタイミングで消えます。
> 予約の追加・変更・削除は、すべて予定マスタのスプレッドシートで行ってください。

### 6.2 Script Properties

- `GITHUB_TOKEN`, `GITHUB_OWNER`, `GITHUB_REPO`, `GITHUB_BRANCH`, `GITHUB_PATH_PREFIX`
- `GCAL_ID_YAKUIN`, `GCAL_ID_JIMUKYOKU`, `GCAL_ID_IINKAI`, `GCAL_ID_CHIKU_HAN`, `GCAL_ID_JISHU`

### 6.3 トリガー

- 編集トリガー: events シート編集後、30秒デバウンスで publishAll
- 時間トリガー: 毎日深夜2時に publishAll(保険)

旧指示書 4.4 と同じ。レイアウト参照は定数オブジェクトに集約し、列追加・順序変更が1箇所で済むようにする。

## 7. やってほしくないこと

- React/Vue/ビルドツール不要。素のHTML+CSS+vanilla JS
- 双方向同期(カレンダー→シート)は実装しない。シートが正、カレンダーは見るだけ
- 差分同期は実装しない。月単位の全削除→全再作成で十分
- 破壊的操作(シート削除、行削除、GitHubファイル一括削除)は実装しない
- 高齢者を想定しないUI(極小フォント、ホバー必須、複雑な操作)はNG
- 「とりあえず動く」コードはNG。JSDocコメント必須、運用後のメンテを意識
